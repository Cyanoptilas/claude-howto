# AI コード生成のためのクリーンコードルール

これらのルールは、保守しやすいプロフェッショナル品質のコードを生成するためのガイドラインです。

## 意味のある名前
- なぜ存在するのかを説明する意図が伝わる名前を使用する
- 誤解を招く名前や無意味な区別を避ける（例：`data`、`info`、`manager`）
- 発音できる・検索できる名前を使用する
- クラス名：名詞（例：`UserAccount`、`PaymentProcessor`）
- メソッド名：動詞（例：`calculateTotal`、`sendEmail`）
- メンタルマッピングやエンコーディング（ハンガリアン記法・プレフィックス）を避ける

## 関数
- 関数を小さく保つ（理想は20行未満）
- ひとつのことだけ行う — 単一責任の原則
- 関数内は一つの抽象レベルに統一する
- 引数は制限する：0〜2個が理想、最大3個、フラグ引数は避ける
- 副作用なし — 関数はその名前が示すことだけを行う
- コマンド（状態変更）とクエリ（情報返却）を分離する
- エラーコードよりも例外を優先する

## コメント
- コードは自己説明的であるべき — コメントはできるだけ避ける
- 良いコメント：法的情報・警告・TODO・公開 API ドキュメント
- 悪いコメント：冗長・誤解を招く・悪いコードを説明するもの
- コードをコメントアウトしない — 削除する（バージョン管理が履歴を保存する）
- コメントが必要なら、代わりにコードのリファクタリングを検討する

## フォーマット
- ファイルを小さく集中させる
- 垂直フォーマット：関連する概念を近くに、空行で概念を分離する
- 水平フォーマット：行の長さを制限する（80〜120文字）
- 一貫したインデントとチームのスタイルを使用する
- 関連する関数をグループ化する

## オブジェクトとデータ構造
- オブジェクト：抽象化の背後にデータを隠し、メソッドを通じて振る舞いを公開する
- データ構造：データを公開し、振る舞いは最小限にする
- デメテルの法則：直接の友人とだけ話す、`a.getB().getC().doSomething()` は避ける
- ゲッター・セッターを通じて内部構造を盲目的に公開しない

## エラーハンドリング
- 例外を使用する（戻り値コードやエラーフラグは使わない）
- コードが失敗する可能性がある場合は最初に `try-catch-finally` を書く
- 例外メッセージにコンテキストを提供する
- `null` を返さない — 空のコレクションを返すか Optional/Maybe を使用する
- `null` を引数として渡さない

## クラス
- 小さなクラス：行数でなく責任で計る
- 単一責任の原則：変更する理由はひとつだけ
- 高凝集：クラス変数が多くのメソッドで使用されている
- 低結合：クラス間の依存関係を最小化する
- 開放閉鎖原則：拡張に対して開いており、変更に対して閉じている

## ユニットテスト
- 高速・独立・再現可能・自己検証・適時（F.I.R.S.T.）
- テストごとにひとつのアサート（またはひとつの概念）
- テストコードの品質はプロダクションコードと同等
- テストの内容を説明する読みやすいテスト名
- Arrange-Act-Assert パターン

## コード品質原則
- **DRY（Don't Repeat Yourself）**: 重複なし
- **YAGNI（You Aren't Gonna Need It）**: 仮定の将来のために構築しない
- **KISS（Keep It Simple）**: 不必要な複雑さを避ける
- **ボーイスカウトの規則**: 見つけたときよりもコードをきれいにして去る

## 避けるべきコードスメル
- 長い関数やクラス
- 重複したコード
- デッドコード（未使用の変数・関数・パラメータ）
- フィーチャーエンビー（他のクラスに興味を持ちすぎるメソッド）
- 不適切な親密さ（お互いについて知りすぎているクラス）
- 長いパラメータリスト
- プリミティブ強迫（小さなオブジェクトの代わりにプリミティブを使いすぎる）
- Switch/case 文（ポリモーフィズムを検討する）
- 一時的なフィールド（時々しか使用されないクラス変数）

## 並行性
- 並行コードを他のコードから分離する
- 同期・ロックされたデータのスコープを制限する
- スレッドセーフなコレクションを使用する
- 同期セクションを小さく保つ
- 実行モデルとプリミティブを理解する

## システム設計
- 構築と使用を分離する（依存性注入）
- 複雑なオブジェクト生成にはファクトリー・ビルダーを使用する
- 実装ではなくインターフェースに対してプログラムする
- 継承よりもコンポジションを優先する
- 単純化するときにのみデザインパターンを適用する（見せびらかすためでなく）

## リファクタリング
- 大きなバッチではなく継続的にリファクタリングする
- 変更の前後で常にテストを通過させる
- 小さなステップ：一度に一つの変更
- 一般的なリファクタリング：メソッド抽出・名前変更・移動・インライン化

## ドキュメント
- 自己文書化コード > コメント > 外部ドキュメント
- 公開 API には明確なドキュメントが必要
- ドキュメントに例を含める
- ドキュメントをコードの近くに保つ（理想的にはコード内に）

---

**核心的な哲学**: コードは書くよりも読む方が10倍多い。機転よりも読みやすさと保守性を最適化する。
