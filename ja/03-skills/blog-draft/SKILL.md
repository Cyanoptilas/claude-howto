## ユーザー入力

```text
$ARGUMENTS
```

処理前にユーザー入力を考慮する必要があります。ユーザーは以下を提供してください：
- **アイデア・トピック**: ブログ投稿の主要コンセプトまたはテーマ
- **リソース**: 調査する URL・ファイル・参照（オプションだが推奨）
- **ターゲットオーディエンス**: ブログ投稿の対象者（オプション）
- **トーン・スタイル**: フォーマル・カジュアル・テクニカルなど（オプション）

**重要**: **既存のブログ投稿**の更新を求めている場合、ステップ0〜8をスキップして**ステップ9**から直接始めます。最初に既存の草案ファイルを読んでから、反復プロセスに進みます。

## 実行フロー

これらのステップを順番に実行します。**指示されたステップをスキップしたり、ユーザーの承認なしに進んだりしないでください。**

### ステップ0: プロジェクトフォルダーの作成

1. `YYYY-MM-DD-short-topic-name` の形式でフォルダー名を生成する
2. フォルダー構造を作成する：
   ```
   blog-posts/
   └── YYYY-MM-DD-short-topic-name/
       └── resources/
   ```
3. 進める前にユーザーとフォルダー作成を確認する

### ステップ1: 調査・リソース収集

各提供されたリソースについて：
- **URL**: 取得してキー情報を `resources/` にマークダウンファイルとして保存する
- **ファイル**: 読んで `resources/` に要約する
- **トピック**: ウェブ検索で最新情報を収集する

各リソース要約に含める：
```markdown
# ソース: [タイトル/URL]

## キーポイント
- ポイント1
- ポイント2

## 関連する引用・データ
- 引用または統計1

## トピックとの関連性
関連性の簡単な説明
```

### ステップ2: ブレインストーミング・明確化

1. アイデアと調査したリソースに基づいて提示する：
   - 調査から特定した**主要テーマ**
   - ブログ投稿の**潜在的な角度**
   - カバーすべき**キーポイント**

2. 明確化のための質問をする：
   - 読者に持ってほしい主なポイントは何ですか？
   - 目標の長さは？（短い: 500〜800語、中程度: 1000〜1500、長い: 2000以上）

3. **続行前にユーザーの回答を待つ。**

### ステップ3〜8: アウトライン・草案・レビュー

- アウトラインを作成して `OUTLINE.md` として保存する
- git リポジトリの場合はコミットする
- 承認されたアウトラインに基づいて草案を書く
- `draft-v0.1.md` として保存する
- レビューのために草案を提示する

**引用要件**: すべてのデータポイント・統計・比較は引用が必要です。

### ステップ9: 反復または最終化

**ユーザーが変更を求めた場合:**
- バージョン番号をインクリメントする（v0.2、v0.3 など）
- すべてのフィードバックを取り込む
- `draft-v[X.Y].md` として保存する

**ユーザーが承認した場合:**
- 最終草案バージョンを確認する
- 必要に応じて `final.md` にリネームする

## バージョントラッキング

すべての草案はインクリメンタルバージョニングで保存されます：
- `draft-v0.1.md` — 初期草案
- `draft-v0.2.md` — 最初のフィードバック後
- `draft-v0.3.md` — 2回目のフィードバック後

## 品質のためのヒント

- **フック**: 質問・驚くべき事実・共感できるシナリオで始める
- **フロー**: 各段落が次につながるようにする
- **証拠**: 調査のデータで主張をサポートする
- **引用**: 統計・データポイント・比較には常に引用する
- **声**: 全体を通じて一貫したトーンを維持する
- **読みやすさ**: 短い段落、適切な箇条書きを使用する
- **CTA**: 明確な行動喚起または思考を促す質問で締めくくる
