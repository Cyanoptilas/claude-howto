## ユーザー入力

```text
$ARGUMENTS
```

ユーザー入力を処理する前に考慮する必要があります（空でない場合）。ユーザーは以下を指定できます：
- `create` - 新しい CLAUDE.md を最初から作成する
- `update` - 既存の CLAUDE.md を改善する
- `audit` - 現在の CLAUDE.md の品質を分析してレポートする
- 作成・更新する特定のパス（例：`src/api/CLAUDE.md`）

## 核心原則

**LLM はステートレスです**: CLAUDE.md はすべての会話に自動的に含まれる唯一のファイルです。コードベースへの AI エージェントの主要なオンボーディングドキュメントとして機能します。

### ゴールデンルール

1. **Less is More**: フロンティア LLM は約150〜200の指示に従えます。Claude Code のシステムプロンプトはすでに約50を使用しています。CLAUDE.md を集中して簡潔に保ちましょう。

2. **普遍的な適用性**: すべてのセッションに関連する情報のみを含めます。タスク固有の指示は別のファイルに置きます。

3. **Claude をリンターとして使わない**: スタイルガイドラインはコンテキストを肥大化させ、指示の追従を低下させます。代わりに決定論的ツール（prettier、eslint など）を使用します。

4. **自動生成しない**: CLAUDE.md は AI ハーネスの最も高いレバレッジポイントです。慎重な検討で手動で作成してください。

## 実行フロー

### 1. プロジェクト分析

まず、現在のプロジェクト状態を分析します：

1. 既存の CLAUDE.md ファイルを確認する：
   - ルートレベル：`./CLAUDE.md` または `.claude/CLAUDE.md`
   - ディレクトリ固有：`**/CLAUDE.md`
   - グローバルユーザー設定：`~/.claude/CLAUDE.md`

2. プロジェクト構造を特定する：
   - 技術スタック（言語・フレームワーク）
   - プロジェクトタイプ（モノレポ・単一アプリ・ライブラリ）
   - 開発ツール（パッケージマネージャー・ビルドシステム・テストランナー）

### 2. コンテンツ戦略（WHAT・WHY・HOW）

CLAUDE.md を3つの次元で構成します：

#### WHAT — 技術・構造
- 技術スタックの概要
- プロジェクト構成（特にモノレポで重要）
- 主要ディレクトリとその目的

#### WHY — 目的・コンテキスト
- プロジェクトが何をするか
- 特定のアーキテクチャ決定が下された理由
- 各主要コンポーネントが担う責任

#### HOW — ワークフロー・規約
- 開発ワークフロー（bun vs node、pip vs uv など）
- テスト手順とコマンド
- 検証とビルドの方法
- クリティカルな「落とし穴」や非自明な要件

### 3. 品質制約

CLAUDE.md の作成・更新時：

1. **目標の長さ**: 300行未満（理想は100行未満）
2. **スタイルルールなし**: リント・フォーマット指示を削除する
3. **タスク固有の指示なし**: 別のファイルに移動する
4. **コードスニペットなし**: 代わりにファイル参照を使用する
5. **冗長な情報なし**: package.json や README と重複しない

### 4. 必須セクション

```markdown
# プロジェクト名

一行の簡単な説明。

## 技術スタック
- 主要言語とバージョン
- 主要フレームワーク・ライブラリ
- データベース・ストレージ（ある場合）

## 開発コマンド
- インストール: `command`
- テスト: `command`
- ビルド: `command`

## 重要な規約
[自明でない・高インパクトな規約のみ]
- 規約1 と簡単な説明
- 規約2 と簡単な説明

## 既知の問題・落とし穴
[開発者がつまずきやすいこと]
- 問題1
- 問題2
```

### 5. 含めてはいけないアンチパターン

**含めないこと：**
- コードスタイルガイドライン（リンターを使用する）
- Claude の使い方に関するドキュメント
- 明白なパターンの長い説明
- コピーペーストされたコード例
- 一般的なベストプラクティス（「クリーンなコードを書く」）
- 特定のタスクへの指示
- 自動生成されたコンテンツ

### 6. 検証チェックリスト

- [ ] 300行未満（できれば100行未満）
- [ ] すべての行がすべてのセッションに適用される
- [ ] スタイル・フォーマットルールなし
- [ ] コードスニペットなし（ファイル参照を使用）
- [ ] コマンドが動作することを確認済み
- [ ] README.md との冗長性なし

## 出力フォーマット

### `create` またはデフォルトの場合：
1. プロジェクトを分析する
2. 上記の構造に従って CLAUDE.md の草案を作成する
3. レビュー用に草案を提示する
4. 承認後に適切な場所に書き込む

### `update` の場合：
1. 既存の CLAUDE.md を読む
2. ベストプラクティスに照らして監査する
3. 以下を特定する：削除すべきコンテンツ・凝縮すべきコンテンツ・不足している必須情報
4. レビュー用に変更を提示する
5. 承認後に変更を適用する

### `audit` の場合：
1. 既存の CLAUDE.md を読む
2. 以下を含むレポートを生成する：現在の行数 vs 目標・普遍的に適用可能なコンテンツの割合・見つかったアンチパターンのリスト・改善のための推奨事項
3. ファイルを変更しない、レポートのみ
