# セキュリティポリシー

## 概要

Claude How To プロジェクトのセキュリティは重要です。このドキュメントでは、セキュリティのプラクティスと、セキュリティ脆弱性を責任を持って報告する方法について説明します。

## サポート対象バージョン

以下のバージョンにセキュリティアップデートを提供しています：

| バージョン | 状態 | サポート期限 |
|---------|--------|---------------|
| 最新版（main） | ✅ アクティブ | 現在 + 6ヶ月 |
| 1.x リリース | ✅ アクティブ | 次のメジャーバージョンまで |

**注意**: 教育ガイドプロジェクトとして、従来のバージョンサポートよりも最新のベストプラクティスとドキュメントセキュリティの維持に注力しています。アップデートは直接 main ブランチに適用されます。

## セキュリティプラクティス

### コードセキュリティ

1. **依存関係管理**
   - すべての Python 依存関係は `requirements.txt` で固定
   - dependabot と手動レビューによる定期更新
   - コミットごとに Bandit でセキュリティスキャン
   - セキュリティチェック用の pre-commit フック

2. **コード品質**
   - Ruff によるリントで一般的な問題を検出
   - mypy による型チェックで型関連の脆弱性を防止
   - pre-commit フックで標準を強制
   - マージ前にすべての変更をレビュー

3. **アクセス制御**
   - `main` ブランチのブランチ保護
   - マージ前に必須レビュー
   - マージ前にステータスチェック通過が必要
   - リポジトリへの書き込みアクセスを制限

### ドキュメントセキュリティ

1. **例にシークレットを含めない**
   - 例のすべての API キーはプレースホルダー
   - クレデンシャルをハードコードしない
   - `.env.example` ファイルで必要な変数を示す
   - シークレット管理についての明確な警告

2. **セキュリティのベストプラクティス**
   - 例ではセキュアなパターンを示す
   - ドキュメントでセキュリティ警告を強調
   - 公式セキュリティガイドへのリンク
   - 関連セクションでクレデンシャルの取り扱いを説明

3. **コンテンツレビュー**
   - すべてのドキュメントをセキュリティ問題についてレビュー
   - コントリビューションガイドラインにセキュリティに関する考慮事項を含める
   - 外部リンクと参照の検証

### 依存関係のセキュリティ

1. **スキャン**
   - Bandit がすべての Python コードの脆弱性をスキャン
   - GitHub セキュリティアラートによる依存関係の脆弱性チェック
   - 定期的な手動セキュリティ監査

2. **アップデート**
   - セキュリティパッチを迅速に適用
   - メジャーバージョンを慎重に評価
   - Changelog にセキュリティ関連のアップデートを含める

3. **透明性**
   - セキュリティアップデートをコミットに記録
   - 脆弱性の開示を責任を持って行う
   - 適切な場合に公開セキュリティアドバイザリを発行

## 脆弱性の報告

### 対象となるセキュリティ問題

以下についての報告を歓迎します：
- スクリプトや例における**コードの脆弱性**
- Python パッケージの**依存関係の脆弱性**
- 任意のコード例における**暗号化の問題**
- ドキュメントにおける**認証・認可の欠陥**
- 設定例における**データ漏洩リスク**
- **インジェクション脆弱性**（SQL、コマンドなど）
- **SSRF/XXE/パストラバーサル**の問題

### 対象外のセキュリティ問題

以下はこのプロジェクトのスコープ外です：
- Claude Code 自体の脆弱性（Anthropic に報告してください）
- 外部サービスやライブラリの問題（アップストリームに報告してください）
- ソーシャルエンジニアリングやユーザー教育（このガイドには該当しません）
- 実証なしの理論的な脆弱性
- 公式チャネルを通じて報告された依存関係の脆弱性

## 報告方法

### プライベート報告（推奨）

**機密性の高いセキュリティ問題には、GitHub のプライベート脆弱性報告をご利用ください：**

1. https://github.com/luongnv89/claude-howto/security/advisories にアクセス
2. 「Report a vulnerability」をクリック
3. 脆弱性の詳細を入力
4. 以下を含めてください：
   - 脆弱性の明確な説明
   - 影響を受けるコンポーネント（ファイル、セクション、例）
   - 潜在的な影響
   - 再現手順（該当する場合）
   - 提案する修正（ある場合）

**その後の流れ：**
- 48時間以内に受信確認
- 調査と重大度の評価
- 修正の開発に協力
- 開示のタイムラインを調整
- セキュリティアドバイザリでクレジット（匿名希望の場合を除く）

### 公開報告

機密性の低い問題またはすでに公開されている問題の場合：

1. `security` ラベルで **GitHub Issue を開く**
2. 以下を含めてください：
   - タイトル：`[SECURITY]` に続けて簡単な説明
   - 詳細な説明
   - 影響を受けるファイルまたはセクション
   - 潜在的な影響
   - 提案する修正

## 脆弱性対応プロセス

### 評価（24時間）

1. 報告受信を確認
2. [CVSS v3.1](https://www.first.org/cvss/v3.1/specification-document) を使用して重大度を評価
3. スコープ内かどうかを判断
4. 初期評価を連絡

### 開発（1〜7日）

1. 修正を開発
2. 修正をレビューおよびテスト
3. セキュリティアドバイザリを作成
4. リリースノートを準備

### 開示（重大度により異なる）

**クリティカル（CVSS 9.0-10.0）**
- 即座に修正をリリース
- 公開アドバイザリを発行
- 報告者に24時間前に通知

**高（CVSS 7.0-8.9）**
- 48〜72時間以内に修正をリリース
- 報告者に5日前に通知
- リリース時に公開アドバイザリ

**中（CVSS 4.0-6.9）**
- 次の定期アップデートで修正をリリース
- リリース時に公開アドバイザリ

**低（CVSS 0.1-3.9）**
- 次の定期アップデートに修正を含める
- リリース時にアドバイザリ

## セキュリティガイドライン（コントリビューター向け）

### 例を書く際

1. **シークレットをハードコードしない**
   ```python
   # ❌ 悪い例
   api_key = "sk-1234567890"

   # ✅ 良い例
   api_key = os.getenv("API_KEY")
   ```

2. **セキュリティへの影響について警告する**
   ```markdown
   ⚠️ **セキュリティ注意**: `.env` ファイルを git にコミットしないでください。
   すぐに `.gitignore` に追加してください。
   ```

3. **セキュアなデフォルトを使用する**
   - デフォルトで認証を有効にする
   - 適用可能な場合は HTTPS を使用する
   - 入力を検証・サニタイズする
   - パラメータ化クエリを使用する

4. **セキュリティに関する考慮事項を文書化する**
   - セキュリティが重要な理由を説明する
   - セキュアなパターンと安全でないパターンを示す
   - 権威ある情報源にリンクする
   - 警告を目立つように含める
